# Metrics Dashboard — Система метрик промптов

> **Level 4: Governed & Measured** — каждый промпт измеряется по 6 ключевым метрикам.
> Детальная документация: [metrics-l4.md](metrics-l4.md)

## Обзор

Dashboard собирает метрики по каждому промпту для отслеживания качества, использования и стабильности.

## Архитектура

```
prompts/<role>/
├── metrics/                    # 🆕 Директория метрик
│   ├── dashboard.md            # Дашборд (aggregated view)
│   ├── usage.md                # Лог использований
│   ├── quality.md              # Оценки качества
│   └── latency.md              # Замеры latency
├── prompt.md
├── card.md
├── test-cases.md
└── changelog.md
```

## Метрики

### 1. Usage Count (Частота использования)

**Где хранится:** `metrics/usage.md`

**Что считаем:**
- Количество обращений к промпту за период
- Количество уникальных пользователей
- Наиболее частые сценарии использования

**Как собирать:**
- Ручная запись после каждого использования (минимум)
- Автоматический сбор через issue/PR references (рекомендуется)
- Отслеживание упоминаний в PR description

**Формат записи:**

```markdown
## 2025-07

| Date       | User        | Scenario              | Source      |
|------------|-------------|-----------------------|-------------|
| 2025-07-12 | @alexe      | API проектирование    | direct      |
| 2025-07-10 | @maria      | Архитектурный аудит   | PR review   |
```

### 2. Test Pass Rate (Проходимость тестов)

**Где хранится:** `test-cases.md` → Metrics → Regression

**Формула:** `passed / total * 100`

**Цели:**
- 🟢 ≥ 95% — отлично
- 🟡 80-94% — внимание
- 🔴 < 80% — критично

**Обновление:** при каждом изменении промпта

### 3. Latency (Время генерации)

**Где хранится:** `metrics/latency.md`

**Что замеряем:**
- Среднее время генерации ответа
- P50, P95, P99 перцентили
- Время по сценариям

**Формат:**

```markdown
## Latency Log

### 2025-07

| Date       | Scenario           | P50  | P95  | P99  | Notes       |
|------------|--------------------|------|------|------|-------------|
| 2025-07-12 | TC-001 (декомпозиция) | 4s  | 6s   | 8s   | —           |
| 2025-07-12 | TC-002 (паттерн)   | 5s   | 8s   | 12s  | сложный кейс |
```

**Цели:**
- 🟢 P50 < 10s
- 🟡 P50 10-20s
- 🔴 P50 > 20s

### 4. Output Quality (Качество вывода)

**Где хранится:** `metrics/quality.md`

**Как оцениваем:**
- Rating 1-5 от пользователя после каждого использования
- Критерии: релевантность, полнота, структура, практическая ценность

**Формат:**

```markdown
## Quality Ratings

### 2025-07

| Date       | User    | Relevance | Completeness | Structure | Value | Avg |
|------------|---------|-----------|--------------|-----------|-------|-----|
| 2025-07-12 | @alexe  | 5         | 4            | 5         | 5     | 4.75|
| 2025-07-10 | @maria  | 4         | 5            | 4         | 4     | 4.25|
```

**Цели:**
- 🟢 Avg ≥ 4.0
- 🟡 Avg 3.0-3.9
- 🔴 Avg < 3.0

### 5. Change Frequency (Частота изменений)

**Где хранится:** `changelog.md`

**Что считаем:**
- Количество коммитов/PR в месяц
- Типы изменений (feat/fix/refactor)
- Размер изменений (строк)

**Формула:** `commits_per_month`

**Интерпретация:**
- 🟢 0-2 PR/мес — стабильный промпт
- 🟡 3-5 PR/мес — активная разработка
- 🔴 > 5 PR/мес — нестабильный, нужна ревью

### 6. Open Issues (Открытые проблемы)

**Где хранится:** GitHub Issues (тег `prompt:python-architect`)

**Что считаем:**
- Количество открытых issues по промпту
- Среднее время решения (days to resolve)
- Приоритет распределения

**Цели:**
- 🟢 < 3 открытых
- 🟡 3-5 открытых
- 🔴 > 5 открытых

## Dashboard (Сводка)

| Метрика            | Значение | Статус | Тренд  |
|--------------------|----------|--------|--------|
| Usage count        | 15       | 🟢     | ↗ +20% |
| Test pass rate     | 100%     | 🟢     | → стаб |
| Latency P50        | 5s       | 🟢     | → стаб |
| Quality Avg        | 4.5      | 🟢     | ↗ +0.3 |
| Changes (this mo)  | 2        | 🟢     | —      |
| Open issues        | 0        | 🟢     | —      |

## Trend (последние 3 месяца)

| Месяц    | Usage | Test% | Latency | Quality | Issues |
|----------|-------|-------|---------|---------|--------|

| 2025-06  | 12    | 100   | 5s      | 4.4     | 2      |

## Статусы

| Статус | Значение |
|--------|----------|
| 🟢 | В норме |
| 🟡 | Требует внимания |
| 🔴 | Критично |
| ↗ | Растёт |
| ↘ | Падает |
| → | Стабильно |
| — | Нет данных |
```

## Сбор метрик

### Ручной сбор (минимум)

После каждого использования промпта:

1. Записать использование в `metrics/usage.md`
2. Если результат нетипичный — оценить качество в `metrics/quality.md`
3. Если замеряли время — записать в `metrics/latency.md`
4. Обновить `metrics/dashboard.md` в конце месяца

### Автоматический сбор (рекомендуется)

1. **Usage:** отслеживать PR references и issue mentions
2. **Test pass:** CI-скрипт `scripts/ci-check.py`
3. **Latency:** замерять через GitHub Actions `time`
4. **Quality:** опросник после использования (Google Forms / GitHub Discussion)

### Quarterly Review

Каждый квартал:
1. Собрать все метрики за квартал
2. Обновить `metrics/dashboard.md`
3. Выявить тренды и аномалии
4. Принять решения: улучшить, деградировать, депривать

## Best Practices

1. **Измеряй регулярно.** Лучше мало данных, чем ничего.
2. **Будь честен.** Записывай и плохие результаты — это данные для улучшения.
3. **Сравнивай с предыдущими периодами.** Тренды важнее абсолютных значений.
4. **Действуй по данным.** Если метрика 🔴 — создай issue для исправления.
5. **Не переусердствуй.** Метрики — инструмент, не цель. Не трать время на сбор ради сбора.
