## Test Cases

### TC-001: Модульная декомпозиция монолита

- **Input:** "Есть монолитный модуль services.py на 800 строк: обработка заказов, уведомления, расчёты скидок. Нужно декомпозировать."
- **Expected:** Предложена декомпозиция на 3 модуля (orders, notifications, discounts) с чёткими интерфейсами и примерами кода
- **Criteria:** Каждый модуль — single responsibility, интерфейсы явные, нет circular dependencies
- **Status:** ✅
- **Notes:** Проверяем что модули не пересекаются по ответственности

### TC-002: Выбор архитектурного паттерна

- **Input:** "Проектируем систему: REST API для каталога товаров + real-time уведомления о статусе заказа + batch обработка nightly отчётов."
- **Expected:** Обоснованный выбор CQRS + Event-Driven для real-time части, REST для каталога, Celery/batch для отчётов
- **Criteria:** Паттерн соответствует типу нагрузки, есть обоснование trade-off
- **Status:** ✅

### TC-003: Управление зависимостями

- **Input:** "Проект с 20+ внешними зависимостями, нет стратегии версионирования, постоянные конфликты при обновлении."
- **Expected:** Предложена стратегия: Poetry с lock-файлом, семантическое версионирование, политики обновления (security / minor / major), dependency isolation
- **Criteria:** Стратегия покрывает все уровни, есть конкретные команды/настройки
- **Status:** ✅

### TC-004: Архитектурные границы в多层ной системе

- **Input:** "Есть система с domain / application / infrastructure слоями. Domain вызывает application, application вызывает infrastructure. Но иногда infrastructure напрямую обращается к domain через глобальные переменные."
- **Expected:** Выявлена проблема circular dependency через globals, предложены решения: DI, interfaces, event-based коммуникация
- **Criteria:** Границы слоёв чёткие, нет обходов через globals, предложены конкретные refactoring steps
- **Status:** ✅

### TC-005: Масштабирование

- **Input:** "Система обрабатывает 1k RPS, ожидается рост до 10k RPS в течение 6 месяцев. Сейчас — single node, synchronous processing."
- **Expected:** План масштабирования: stateless application layer, read replicas для БД, caching (Redis), async queue (Celery/RQ), горизонтальное масштабирование
- **Criteria:** Решение покрывает bottleneck'ы на каждом уровне, есть приоритизация шагов
- **Status:** ✅

### TC-006: Рефакторинг с сохранением обратной совместимости

- **Input:** "Нужно переписать legacy API с Django на FastAPI, но старый API используется 3 внешними клиентами. Нельзя сломать совместимость."
- **Expected:** Предложена strategy pattern: dual-run (старый + новый), API gateway с роутингом, постепенный migration, contract testing
- **Criteria:** Обратная совместимость сохранена, migration incremental, есть rollback plan
- **Status:** ✅

### TC-007: Запрос вне зоны ответственности

- **Input:** "Напиши мне фронтенд на React для моего Python бэкенда, и настрой Docker Compose с Nginx."
- **Expected:** Вежливый отказ с объяснением что это вне зоны ответственности, предложение альтернатив (API-first дизайн рекомендации, архитектурные аспекты Docker)
- **Criteria:** Чёткое объяснение границ, альтернатива релевантна
- **Status:** ✅
